Um codesmell (cheiro no código - tradução literal) é uma indicação superficial que na maioria das vezes corresponde a um problema mais profundo no sistema. O termo foi cunhado por Kent Beck enquanto me ajudava com o meu livro de refatoração (Refactoring Book / Refatoração ).

A definição rápida acima contém alguns pontos sutis. Em primeiro lugar, um smell (cheiro) é, por definição, algo que é rápido de detectar - fedorento (ou de forte odor), como eu disse recentemente. Um método longo é um bom exemplo disso - basta olhar para o código e meu nariz se contorce se eu ver mais de uma dúzia de linhas de java.

A segunda é que smells nem sempre indicam um problema. Alguns métodos longos são bons. Você tem que se aprofundar no código para ver se há um problema inplicito/encoberto nele - smells não são inerentemente ruins por conta própria - eles são muitas vezes um indicador de um problema e não o problema em si.

Os melhores smells são de fácil detectção e, na maioria das vezes, levam a problemas realmente interessantes. Classes de dados (classes com todos os dados e sem comportamento) são bons exemplos disso. Você olha para elas e se pergunta: qual comportamento essa classe deve ter ? Então você começa a refatorar para mover esse comportamento para lá. Freqüentemente, perguntas simples e refatorações iniciais podem ser o passo vital para transformar objetos anêmicos em algo que realmente tenha classe.

Uma das coisas boas dos smells é que é fácil para pessoas inexperientes detectá-los, mesmo que elas não saibam o suficiente para avaliar se há um problema real ou para corrigi-los. Já ouvi falar de lead developers (desenvolvedores líderes) que escolhem um "smell da semana" e pedem que as pessoas procurem o smell e leve-os aos membros mais experientes da equipe. Fazer um smell de cada vez é uma boa maneira de ensinar gradualmente as pessoas da equipe a serem melhores programadores.
